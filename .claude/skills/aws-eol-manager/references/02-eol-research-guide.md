# 02 — EOL調査ガイド・優先度判定基準

## 調査手順

### 1. AWS公式情報の検索

各サービスについて以下の検索を実施する:

```
検索クエリパターン:
1. "[サービス名] end of life AWS [年]"
2. "[サービス名] end of support AWS"
3. "[サービス名] version policy AWS"
4. "[サービス名] migration guide AWS"
5. "AWS [サービス名] deprecated"
```

### 2. 主要な情報源

| 情報源 | URL パターン | 内容 |
|--------|-------------|------|
| AWS公式ドキュメント | docs.aws.amazon.com | サービス別のバージョンポリシー・移行ガイド |
| AWS公式ブログ | aws.amazon.com/blogs/ | EOLアナウンス・移行ベストプラクティス |
| AWS Health Dashboard | health.aws.amazon.com | サービス別の通知・アナウンス |
| AWS What's New | aws.amazon.com/about-aws/whats-new/ | 新バージョンリリース・EOL告知 |
| AWS ナレッジセンター | repost.aws | 移行に関するQ&A・トラブルシューティング |

### 3. サービスごとの確認ポイント

#### RDS（MySQL / PostgreSQL / MariaDB 等）
- RDS バージョンポリシーページでEOL日を確認
- 延長サポート（Extended Support）の有無と追加コスト
- メジャーバージョンアップグレードの手順（スナップショット→リストア or インプレース）
- パラメータグループ・オプショングループの互換性

#### Amazon Linux / EC2
- Amazon Linux 2: 2025-06-30 EOL（標準サポート終了）
- AL2023への移行ガイド
- カーネル・パッケージの互換性確認

#### EKS（Elastic Kubernetes Service）
- Kubernetesバージョンのサポートポリシー（標準サポート14ヶ月）
- 延長サポート（最大26ヶ月、追加料金あり）
- クラスターアップグレードの手順と注意点

#### Lambda ランタイム
- ランタイムのサポートポリシー（Phase 1 / Phase 2 の非推奨化）
- Phase 1: 新規作成不可、Phase 2: 更新不可
- ランタイムバージョンの移行手順

#### ElastiCache
- Redis / Memcached のバージョンサポートポリシー
- エンジンバージョンのアップグレード手順

#### OpenSearch Service（旧 Elasticsearch Service）
- エンジンバージョンのサポートポリシー
- ドメインのアップグレード手順（ブルー/グリーンデプロイ）

## 優先度判定基準

### 高（赤）
以下のいずれかに該当する場合:
- EOL期限まで **90日未満**
- セキュリティパッチの提供が既に停止
- 延長サポートが **ない**、またはすでに終了
- 本番環境で使用中のリソース
- コンプライアンス要件に抵触する可能性

### 中（黄）
以下のいずれかに該当する場合:
- EOL期限まで **90〜180日**
- 延長サポートを利用中（ただし追加コスト発生中）
- ステージング・検証環境で使用中
- 移行の複雑度が高い（大規模データ移行・アプリケーション変更が必要）

### 低（緑）
以下のいずれかに該当する場合:
- EOL期限まで **180日以上**
- 延長サポートの残り期間が十分にある
- 開発・テスト環境のみで使用
- 移行が比較的容易（マイナーバージョンアップのみ）

---

## 移行複雑度の判定

### 高
- メジャーバージョンアップグレードが必要
- アプリケーションコードの変更が必要（API非互換等）
- 大規模なデータ移行が伴う
- ダウンタイムが長時間必要（数時間〜）
- 他のサービスとの依存関係が多い

### 中
- マイナーバージョンアップグレードで対応可能
- 設定変更・パラメータ調整が必要
- 短時間のダウンタイムが必要（数分〜1時間）
- 一部の設定・テンプレートの更新が必要

### 低
- パッチバージョンアップまたは自動アップグレード対応
- アプリケーション変更不要
- ダウンタイムなし、またはローリングアップデート対応
- AWS側で自動移行される場合

## 推奨スケジュール策定ルール

EOL期限を基準に、以下のマイルストーンを逆算する:

```
EOL期限 ─────────────────────── D-Day
  │
  ├── 3ヶ月前 ── 調査完了・移行計画策定
  │
  ├── 2ヶ月前 ── 検証環境での移行テスト開始
  │
  ├── 1ヶ月前 ── 本番環境での移行実施
  │
  └── 2週間前 ── 移行完了・動作確認・クリーンアップ
```

### 延長サポートがある場合
- 延長サポート終了日をD-Dayとして再計算
- ただし追加コストが発生するため、標準EOL日までの対応を推奨

### 複数サービスが同時期にEOLの場合
- 依存関係のあるサービスは同時に移行する
- 独立したサービスは並行作業で効率化する
- リスクの高い（本番環境・高優先度）サービスを先に対応する
